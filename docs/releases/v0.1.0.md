# pkarena0-web v0.1.0

**Released:** 2026-04-25
**Engine:** [`pkcore`](https://crates.io/crates/pkcore) `0.0.50`
**Play it:** <https://imperialbower.github.io/pkarena0-web/>

---

## Overview

`pkarena0-web` is a browser-based, single-player No-Limit Texas Hold'em
table where one human plays against eight AI bots. The game engine is
written in Rust, compiled to WebAssembly, and served as a single static
HTML page — no server, no account, no install.

`v0.1.0` is the first tagged `0.1.x` release. It consolidates everything
shipped during the `0.0.x` prototype phase, plus the cleanup work that
unblocked the version bump. This is still pre-1.0 software: the public
API surface (such as it is — a handful of WASM exports) and the YAML
hand-history format may change before `1.0`.

## Highlights

- **Single-player NLHE vs. eight bots**, deployed as a static WASM page
  on GitHub Pages.
- **Arena (all-bot) mode** alongside the human Play tab, so you can
  watch the bots play each other.
- **Settings panel** with persisted preferences (sound off by default).
- **Phase 1 voice narration** (experimental) using the browser's
  `SpeechSynthesis` engine.
- **YAML hand-history export** for the entire session.
- **End-to-end Playwright tests** running on every PR.

## What's new

### Gameplay

- Eight-bot table with bots drawn from `pkcore`'s profile pool. The
  pool was upgraded mid-cycle for stronger play (#6).
- Hand log now shows **every player's hole cards**, not just yours
  post-showdown (#7).
- **Per-suit card colouring** in the SVG table makes hand reading
  noticeably faster (#8).
- Fixed blinds at $50 / $100; you start each session with $10,000.

### UI

- New **Arena** tab to run an all-bot table; tab switching cleanly
  halts the loop in the inactive tab (#3, #4).
- **Settings gear** in the score bar with a persisted sound toggle
  (#4, #5).
- Arena score bar shows only the hand number + settings gear — chips
  and P&L are hidden because they are meaningless without a human
  player.
- Score bar shows the running version as a clickable link back to the
  GitHub repo (powered by a new `version()` WASM export).

### Audio (experimental)

- A Phase 1 audio layer ships in `www/audio/` (#5):
  - Voice clip stitcher with `SpeechSynthesis` TTS fallback.
  - `LiveAdapter` that polls `get_state()`, diffs snapshots, and emits
    `GameEvent`s to the audio layer.
  - Voice narration toggle and a "Test voice" button in the settings
    panel.
- Pre-recorded voice clips are **not yet bundled** — the layer falls
  back to TTS for now. See [`docs/audio-integration.md`](../audio-integration.md)
  and [`docs/voice-script.md`](../voice-script.md) for the design and
  the clip script.

### Hand history

- **Export YAML** button downloads a YAML hand history of the entire
  session, suitable for offline replay or analysis.

### Engineering

- `version()` WASM export returning `CARGO_PKG_VERSION`.
- Playwright end-to-end test suite (game flow, UI, YAML download)
  running on every PR via GitHub Actions.
- `Cargo.lock` is now tracked, pinning the dependency graph for
  reproducible release builds and stable CI cache keys.
- `CHANGELOG.md` added; see [`CHANGELOG.md`](../../CHANGELOG.md) for
  the structured changelog format.
- `pkcore` bumped iteratively from `0.0.43` → `0.0.50`, picking up
  betting fixes, leak fixes, and bot improvements along the way.

## Bug fixes

- Three-way pot betting bug where a `call` followed by a re-raise
  blocked the player.
- Even-split bug after a player busts out.
- Tab switching could leave a stale animation loop running.
- Blinds wiring during the human Play loop.
- Chrome `SpeechSynthesis.cancel()` regression: removed the cancel
  call from `voice.cancel()` and the test handler; added a 50 ms TTS
  debounce.
- `cargo check` / `cargo build` no longer fail on a missing
  `validate-yaml` binary stanza in `Cargo.toml`. The corresponding
  `make test-yaml` target has been removed until the validator is
  reinstated.
- README now lists the same minimum Rust toolchain as `Cargo.toml`
  (`1.94.1`).

## Known issues

### Game state resets when the page is backgrounded (mobile)

If a mobile OS evicts the backgrounded tab to reclaim memory, the
browser later restores the tab by reloading the page. Because the
entire game state lives in Rust `thread_local!` singletons inside the
WASM module — and nothing is currently persisted to `localStorage` —
the in-progress hand and session are lost on restore.

- **Severity:** High on mobile, low on desktop.
- **Workaround:** None. Use the **Export YAML** button periodically if
  you want to preserve hand history.
- **Status:** Deferred. A fix likely depends on upstream `pkcore`
  changes (snapshot/restore for `PokerSession`).

Full triage notes — symptom, repro, root cause, fix candidates — live
in [`docs/known-issues.md`](../known-issues.md).

## Compatibility & requirements

| | |
|---|---|
| Rust toolchain | ≥ 1.94.1 with `wasm32-unknown-unknown` target |
| Build tool | [`wasm-pack`](https://rustwasm.github.io/wasm-pack/) |
| Browsers | Modern Chromium / Firefox / Safari with WASM support |
| Engine | [`pkcore`](https://crates.io/crates/pkcore) `0.0.50` |
| License | GPL-3.0-or-later |

There is no upgrade path from `0.0.x` — game state has never been
persisted, so each release simply replaces the previous one in the
browser cache.
